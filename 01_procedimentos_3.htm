Fechar âmbito

 

 

Descrição Sumária:

Este procedimento tem como objectivo garantir a validação e aceitação do âmbito do Projeto de Desenvolvimento do Produto / Release.

 

Regras:

1. Identificação clara do(s) Produto(s) que irão ser alvo de alterações

2. Nº de utilizadores, geografias e locais onde o Projeto de Desenvolvimento do Produto / Release terá impacto

3. Processos de negócio a analisar e rever

4. Requisitos funcionais a cobrir

5. Requisitos técnicos a cobrir

6. Aplicações a alterar e a construir

7. Plataformas e ferramentas a utilizar

 

Acções:

1. Detalhar o âmbito da release

O âmbito da Release/iteração deve ser re-planeado com base nos esforços estimados para cada caso de uso e requisitos descritos como mandatórios, sejam funcionais ou técnicos. Não apenas o esforço estimado para as tarefas de desenvolvimento devem ser fixados mas também todos os demais esforços como:

- Refinamento das especificações funcionais e técnicas

- Testes

- Configurações

Os requisitos envolvidos devem ser listados bem como as mudanças aprovadas a serem realizadas.

 

2. Definir organização da Release / Iteração

O Gestor de Produto deve re-fazer a proposta de divisão do trabalho com base no perfil da equipa.

 

3. Detalhar cronograma da Release / Iteração

O cronograma deve ser revisto com base na visão mais alargada do âmbito a entregar.

 

 

Entregáveis:

1. Documento de Visão da Release (Revisão final para aprovação e validação)

 

nota: A obrigatoriedade de entrega deste documento está identificada na Matriz de Obrigatoriedade de Documentação

Qualificar a Release

 

 

Descrição Sumária:

Este procedimento tem como objectivo garantir a qualificação do Projeto de Desenvolvimento do Produto / Release para servir de orientação ao Gestor de Produto na preparação dos trabalhos de arranque da fase de Execução da Framework de Gestão de Produto.

 

Regras:

1. Documento de visão da Release aprovado ou validado pela equipa de gestão da UN Transportes

2. Proposta comercial adjudicada / change request aprovado

 

Acções:

1. Abrir documento Matriz de Qualificação

O objetivo desta matriz é sinalizar para determinada Release / Iteração a dimensão, complexidade e risco da mesma.

 

2. Preenchimento

O Gestor de Produto deverá ter na sua posse os seguintes elementos:

- Esforço previsto e dimensão da equipa prevista

- Duração prevista

- Custo previsto

 

E finalmente, de acordo com a experiência e histórico de releases anteriores, deverá preencher as células correspondentes ao seu entendimento da complexidade da Release / iteração.

 

 

3. Interpretação dos resultados

As Releases foram classificadas em 5 niveis de acordo com a sua dimensão e complexidade.

 

O que se pretende com esta qualificação é ajudar o Gestor de Produto e equipas participantes com a necessidade de mecanismos de controlo  para se garantir o propósito da Release e ao mesmo tempo balancear a carga administrativa na elaboração de documentos no contexto da execução da Release.

 

Após obter o nível da qualificação, o Gestor de Produto deverá confrontar esse resultado com a Matriz de Obrigatoriedade de Documentação e partilhar esses resultados com a equipa prevista da Release.

Definir estratégia de Garantia e Controlo de Qualidade da Release

 

 

Descrição Sumária:

Este procedimento tem como objectivo que se garanta uma estratégia de garantia e controlo de qualidade da release e os objetivos das actividades de testes.

A estratégia define as ferramentas, técnicas e tipos de teste a serem empregadas.

 

Regras:

1. Uma estratégia de Garantia e Controlo de Qualidade é uma abordagem geral para o processo de teste que permita o aumento da confiança da equipa e dos vários stakeholders no produto final a ser entregue. Podem ser adoptadas estratégias diferentes dependendo do produto a ser testado e o processo de desenvolvimento utilizado (ex: cascata ou agile).

 

 

Acções:

1. ESTUDO DOS REQUISITOS

• Estudar os requisitos funcionais e não funcionais solicitados pelo Cliente (proposta comercial adjudicada ou change request aprovado);

• Estudar as modificações de requisitos solicitados pelo projeto / release (mudanças de requisitos);

• Revisão da PBS e identificar “inconsistências” dos requisitos;

• Analisar os documentos formalmente adjudicados e “feedback” do entendimento do âmbito estabelecido dos mesmos;

• Avaliar expectativas de custos, prazos e qualidade exigidas pelo Cliente;

• Avaliar os riscos envolvidos nos Projetos e seus impactos neste processo;

 

2. AVALIAÇÃO DE IMPACTO

• Avaliar se o projeto exige a criação de casos de testes “não funcionais”;

• Avaliar se o projeto exige modificações em casos de testes “regressivos”

• Avaliar se o projeto exige automação dos testes;

• Avaliar se o projeto exige adequação nas atuais ferramentas empregadas;

• Avaliar se o projeto exige a aquisição/construção de novas ferramentas;

• Avaliar se o projeto exige modificações na estruturação do ambiente;

• Inventariar Lista de Projetos em Curso e a serem Iniciados;

• Avaliar a disponibilidade de recursos internos para alocação no Projeto / Release;

• Identificar Cenários para alocação ao Projeto / Release (Terceirização, Redução de Âmbito de Garantia e Controlo de Qualidade, Prioritização de Projetos);

• Definir Cronograma-Macro para cada cenário identificado;

• Definir Riscos para cada cenário identificado e Planos de Ação Esperados;

• Estabelecer Proposta de Plano V&V para o Projeto / Release;

• Levantar métricas históricas para auxiliar na elaboração das estimativas de esforço;

• Estimar esforço interno para absorção dos impactos da Framework V&V;

• Demonstrar esforço externo para absorção dos impactos da Framework V&V;

 

3. ESPECIFICAR AS ADAPTAÇÕES NA FRAMEWORK V&V

• Especificar as adequações nas atuais ferramentas empregadas pela equipa V&V;

• Especificar as novas ferramentas exigidas pelo projeto / release;

• Especificar as modificações estruturais na organização do ambiente de Qualidade;

• Especificar as adequações nas baterias de testes atuais;

 

4. ELABORAR PLANO V&V

• Definição de um Plano de V&V suportado nas expectativas do Projeto / Release em relação a prazos, custos e qualidade esperada

• Assegurar o dimensionamento da equipa e estabelecer um esforço de acordo com as necessidades apontadas pelo Projeto / Release.

• Instanciação da Framework V&V de acordo com o âmbito e qualificação definido

• Assegurar o grau de cobertura necessário para a qualidade esperada pelo Projeto / Release

• Definição de um Plano de Controlo e Métricas que permita aferir a qualidade do Projeto / Release

• Assegurar a gestão de requisitos

 

 

Entregáveis:

1. Plano V&V (Incorpora estratégia de garantia e controlo de qualidade no Documento de Visão da Release (Revisão final para aprovação e validação)
